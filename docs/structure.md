## Intended Documentation Layout

- `AGENTS.md`: instructions for AI coding agents and future contributors.
- `README.md`: short entry point for humans landing in the repository.
- `examples/`: sample imported Hugo-style content used to understand front matter and migration needs.
- `docs/content.md`: content model, front matter, routing, and migration compatibility notes.
- `docs/intent.md`: product purpose, values, use cases, and open product decisions.
- `docs/structure.md`: repository organization and file ownership expectations.
- `docs/tools.md`: local commands, dependencies, verification, and deployment tooling.

## Structural Guidelines

- Keep content files easy to read outside the application.
- Treat `examples/` as reference material for imports and front matter compatibility, not as production content.
- Keep generated files out of the main source tree unless the chosen tool requires otherwise.
- Put reusable rendering logic in `templates/` or Go packages, not beside individual posts.
- Keep `dist/` disposable; it should be reproducible from source content, templates, static assets, and Go code.
- Generate blog post URLs under `/posts/`.
- Support generated pages and tag listing pages as first-class output types.
- Put project decisions in `docs/`, not in scattered comments.
- Keep top-level directories few and obvious.
- Add React-specific directories only after an accepted decision to add React.

## Open Structure Decisions
